快速理解

一句话解释：gdt-server 把 openLCA 数据库和计算引擎包装成 HTTP 接口。调用方先查到模型 UUID，再发起计算，使用返回的临时结果 ID 查询各种结果， 最后主动释放结果。
1. 查 Product System / Method UUID→ 2. POST /result/calculate→ 3. GET /state→ 4. 查总量与贡献→ 5. POST /dispose

当前 PCF 项目重点使用

版本检查、描述符/参数查询、计算、状态、总影响、过程贡献和结果释放。 API 卡片左侧蓝线及“PCF 核心”标签表示该范围。

结果是临时资源

POST /result/calculate 返回的 @id 是结果句柄，不是永久业务 ID。 查询结束必须调用 dispose；本项目在 finally 中执行释放。

只读模式

当前项目用 --readonly 启动 gdt-server，因此 PUT、DELETE 和 /data/create-system 根本不会注册，应在 Desktop 中完成模型编辑。

路径参数速查

参数格式通俗说明
{id}UUID / 结果句柄 /data 下表示实体 UUID；/result 下表示计算返回的临时结果 ID。
{type-path}模型类型路径 告诉服务器要查询哪类 openLCA 对象，允许值见下表。
{tech-flow}provider UUID::flow UUID 技术流唯一标识，即“哪个过程提供哪条产品流”。应先从 /result/{id}/tech-flows 返回值取得。
{envi-flow}flow UUID 或 flow UUID::location UUID 环境流标识，例如某地点的 CO₂ 排放流。应先从 /result/{id}/envi-flows 取得。
{impact-category}影响类别 UUID 例如 Climate change/GWP 类别 ID，应从计算结果的 impact-categories 或 total-impacts 取得。
<name>URL 编码后的对象名称 按名称查对象。名称可能不唯一，自动化集成优先使用 UUID。

共享请求体模型

Ref：对象引用

最常用字段是 @id、@type、name。 计算时 target、impactMethod、unit、context 等均使用引用。

ParameterRedef：参数重定义

name 参数名，value 新值，context 用于区分同名参数属于哪个过程或产品系统。

CalculationSetup：计算配置

必填核心是 target；PCF 通常还传 impactMethod 和 parameters。

SankeyRequest：桑基图配置

用 impactCategory、enviFlow 或 forCosts 选择指标，maxNodes 和 minShare 控制图规模。

CalculationSetup 字段

字段类型/可选值作用
targetRef计算目标，通常是 Product System 描述符。
impactMethodRef影响评价方法；没有它就无法查询 LCIA 影响结果。

兼容性、安全与响应约定

安全提醒：该固定版本没有内置用户认证， 并配置为允许任意 CORS 来源。生产环境不要直接暴露到公网；应限制 Worker 网络访问， 或在前方使用带鉴权、TLS、限流和审计的反向代理。
客户端/服务端差异： 当前 olca-ipc 2.6.3 中的 put_source_file、 grouped result 等方法，在该固定服务端源码中没有对应路由；不要将它们当成本服务可用 API。 相反，/result/{id}/total-cost-values 在服务端存在，但 Python 客户端没有直接封装， 需要直接发 HTTP 请求。/results/* 仅用于兼容旧版本，新代码统一使用 /result/*。
HTTP 状态含义处理建议
200成功，通常返回 JSON。按接口响应结构解析。
400路径参数、模型类型或 JSON 请求体错误。检查 UUID 格式、type-path 和请求体。
404对象或结果为空/不存在。确认数据库、UUID、结果是否已被释放或超时清理。
500数据库、模型链接或计算内部错误。查看 gdt-server 日志并核对模型。
